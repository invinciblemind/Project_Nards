Архитектура
===========

Проект строится вокруг строгого разделения слоёв:

* ``nardy.app`` — композиция приложения, запуск, orchestration и сценарии;
* ``nardy.domain`` — чистая предметная модель, правила, состояние партии и движок;
* ``nardy.ui`` — tkinter-представление и пользовательские сценарии;
* ``nardy.net`` — транспорт и сетевые адаптеры для будущего LAN-режима;
* ``nardy.i18n`` — перевод интерфейса и инфраструктура локализации.

Ключевые ограничения:

* tkinter не должен появляться в ``domain``;
* сокеты не должны проникать в ``domain``;
* undo должен работать со снимками состояния игры;
* выбор режима игры должен происходить на уровне приложения и правил, а не через ветвления внутри GUI.

Стратегия расширения
--------------------

Каркас проектируется так, чтобы локальная реализация и сетевая реализация использовали одинаковую доменную модель и общий движок. Для этого UI и транспорт должны общаться с приложением через чёткие команды и DTO, а не напрямую менять состояние.
